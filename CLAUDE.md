# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Super Mario Kart 3** — a 3D Mario Kart-style racing game built on **Three.js r128**, served as a **zero-build, classic-script** web app. There is no bundler, no `node_modules` used at runtime, and no TypeScript/React despite what the README's "stack" section claims — the actual playable game is plain ES5/ES6 scripts in `client/public/`.

This folder (`super-mario-kart3`) is the **upgraded successor copied from `../super-mario-kart2`**: 6 courses (adds Koopa Troopa Beach + Kalimari Desert with a level-crossing steam train), new enemies (Crab, Cheep-Cheep, Pokey, Tumbleweed, Train), new items (Fake Item Box, Golden Mushroom), boost pads, slipstream drafting, themed wheel spray, driver/kart animation upgrades, and final-lap tempo + finish confetti.

## Repository

## Commands

```bash
npm start          # = node server/index.js  → http://localhost:5173/game.html
npm run dev        # identical (alias)
```

`server/index.js` is a dependency-free Node static server that serves `client/public` (root `/` → `game.html`, unknown paths fall back to `game.html`).

### Verifying changes (no test framework)

There is no test runner or linter configured. Smoke-check syntax after edits:

```bash
node --check client/public/js/<file>.js      # one file
for f in client/public/js/*.js; do node --check "$f"; done
```

`node --check` only validates syntax (it cannot run the game — every module references the global `THREE`). To exercise runtime logic headlessly, scripts can be `vm.runInThisContext`-loaded in order against a `THREE`/`window`/`document` mock; nothing depends on a real WebGL context until `renderer.render` is called.

## Architecture

### The `window.MK` namespace + ordered script loading (read this first)

Everything lives on one global object, `window.MK`. Each file is an IIFE — `(function (MK) { 'use strict'; ... })(window.MK)` — that attaches classes (`MK.Kart`), singletons (`MK.audio`, `MK.input`, `MK.hud`, `MK.ui`, `MK.game`), or data (`MK.CONFIG`, `MK.CHARACTERS`, `MK.ITEMS`, `MK.COURSES`). **There are no imports.** Cross-module access is always `MK.*`.

`game.html` loads the 17 scripts in a **fixed dependency order** — changing it breaks `class X extends MK.Y` and top-level data references (most runtime cross-calls are order-independent, but class inheritance and load-time data are not):

```
config → utils → audio → input → particles → characters → kart → items →
scenery → track → ai → camera → hud → courses → ui → game → main
```

When adding a module: wrap it in the IIFE pattern and insert its `<script>` in `game.html` at the right point in this chain.

### `Game` is the "world"

`MK.Game` (`game.js`) owns the renderer/scene/camera, the state machine, and the per-frame loop. **The `Game` instance is passed as `world` into `Kart`, `ItemSystem`, and `AIController`**, which reach back through it: `world.scene`, `world.particles`, `world.items`, `world.track`, `world.karts`, `world.player`, plus callbacks `world.onLightning(owner)` and `world.shake(amount)`. To understand any system you usually need `game.js` open alongside it.

State machine: `title → charselect → courseselect → countdown → racing → finished → results` (+ `paused`). `main.js` wires DOM-menu callbacks (`MK.ui.cb.*`) to `Game` methods. The UI (`ui.js`) only handles menu screens; `Game` owns the transitions that must tear down a race (`quitToTitle`, `toCourseSelect`, `restart`, via `cleanupRace`/`_buildRace`). Per-frame work is split by state in `Game.frame` → `_updateCountdown` / `_updateRacing` / `_updateFinished`; menus/pause render only. Menu screens are **keyboard-navigable** (`UI._onMenuKey`, a global keydown): arrow keys move the `.char-card`/`.course-card` grid selection (column count inferred from `offsetTop`), Enter confirms/advances, Esc/Backspace goes back — alongside mouse clicks. On **touch devices** the on-screen pad doubles as menu input via the `InputManager.onTouch(role, isDown)` hook (`UI._onTouchButton`): `◀`/`▶` move the cursor and `★` confirms/starts. `InputManager.setTouchMode('menu'|'race')` toggles a class on `#touch-controls` so the driving-only buttons (`DRIFT`/`ACCEL`, marked `.race-only`) hide in menus; the racing pad is `◀▶` steer · `★` item · `DRIFT` · `ACCEL` (the accelerator — renamed from the unclear "GAS"). `UI.show*` drives `showTouch`/`setTouchMode`.

### Coordinate & heading conventions (used everywhere)

- **Forward = `-Z`.** `group.rotation.y = yaw`; `forward = (-sin yaw, 0, -cos yaw)` (see `Kart.forward`).
- **Steering:** `steer > 0` means right, which *decreases* yaw.
- **Track normal points LEFT.** `track.project(pos).lateral = (pos - point) · normal`, so positive lateral = left of the centerline.

Kart, track, AI, and camera all assume these. Break one and motion goes wrong silently.

### Track model (`track.js`)

A course centerline is a closed `THREE.CatmullRomCurve3` built from `course.points`. It is sampled into `track.samples[i] = {point, tangent, normal}` at uniform arc-length (`getPointAt`/`getTangentAt`). Road, curbs, and (optionally) walls are generated as ribbon `BufferGeometry`. `track.project(pos)` does a global nearest-sample search and returns `{index, fraction, lateral, point, tangent, normal}`.

Two important contracts:
- `project()` returns **references to the sample's shared vectors** (`point/tangent/normal`). Do not mutate them, and consume the result before the next `project()` call.
- **`project(pos, hint)` is the fast path.** Given a `hint` sample index it only searches a ±40 window around it (else it falls back to the full global scan). Continuously-moving callers pass their last index so it's O(1) instead of O(N): karts pass `this.sampleIndex` (and re-acquire with a *hintless* full scan after a fall/respawn — see `kart.js`), projectiles carry `pr._si`, hazards pass `hz._baseIndex`. Only seed a hint when motion is continuous; a teleport must use the hintless form or the window can lock onto the wrong (parallel) corridor.
- **Lap counting lives in `Kart._updateProgress`**, not the track: it watches the fraction wrap (`>0.85 → <0.15`) and only counts it after a half-lap checkpoint (`passedHalf`) was crossed. Karts must be initialized with a correct `sampleIndex` and `lastFraction` after placement (see `Game._buildRace`).

### Kart physics (`kart.js`)

Arcade model: a scalar `speed` advanced along `yaw`. Controls are pushed in from outside as `controls = {throttle, brake, steer, drift, reverseHeld}` — the **player fills them from `MK.input`, the AI fills the same struct from `AIController`** (`ai.js`). Drift accumulates a charge that crosses `CONFIG.miniTurbo` tiers (blue/orange/purple) and fires `applyBoost` on release. Hit reactions (`spinOut` / `squish` / `launch` / `giveStar`) are gated by `isHittable()`. Visual transform hierarchy: `group` (yaw + world position) → `tilt` (drift roll + hop) → `chassis` (+ driver model). Shadows are fake blob sprites, not real shadow maps. Kart-vs-kart contact pushes a decaying lateral `kart.bumpVel` (set in `Game._resolveCollisions` from the pair's relative speed + weights, integrated then damped each `update` before the wall/void check) so bumps visibly knock racers off their line — lighter karts get bounced and slowed more, heavier ones push through.

**Per-frame race extras live in `Kart._updateRaceExtras`** (called from `update` after `_updateProgress`):
- **Boost pads** — `track.boostPads` entries are `{index, lat, halfW, lenSamples}`; a kart within the sample window and lateral band gets `applyBoost(C.padBoost)` with a per-kart `_padCD` (1.1s) so one pad fires once. Pad ribbons are built in `Track._buildBoostPads` from `course.boostPads` (`{f, lat(-1..1), w}` fractions); the chevron canvas texture points toward +v = direction of travel.
- **Slipstream** (`draftT`) — tailing a kart `2.5..C.draftDist` units ahead (arc-length via `progress * track.length`) with small lateral/yaw deltas accumulates `draftT`; at `C.draftTime` it fires a pale-blue `applyBoost(C.draftBoost)` and resets to a negative cooldown. Wind-streak particles (`draftStreak`) show while accumulating.
- **Golden Mushroom** — `items.js` sets `kart.goldenTimer = C.goldenDuration` and boosts on every press *without consuming* the item; the timer is decremented here and the item slot is cleared when it expires.

**Visual upgrades:** the chassis is scaled by weight class (`_sizeScale = 0.92 + (weight-1)*0.045` — star-blink `chassis.scale` and the blob-shadow scale must multiply by it, not reset to 1), the hood roundel carries the driver's initial (`buildChassis(color, accent, letter)`), drivers lean/turn into corners (driver group + `userData.head` yaw driven in `_updateVisual`), finished karts do a bouncing victory loop, and `_emit` sprays **theme-colored** kickup (snow white / beach+desert sand / castle ash / rainbow sparkle; drifting on snow/sand kicks spray even on-road). **Drivers grip the steering wheel:** the chassis wheel is a spoked rim group at `(0, 1.06, -0.42)` tilt `x=1.22` exposed as `chassis.userData.steerWheel` and **counter-rotated with steering** in `_updateVisual`; every character builder reaches its hands to that point — `humanoidBody` takes a driver-local `opts.grip` `[x, y, z]` (≈ `[0.26, 0.36, -0.77]` divided by the character's root scale, e.g. Wario `[0.22, 0.37, -0.65]`) and computes arm length/pitch from it, while Peach/Toad/Yoshi/DK/Bowser have hand positions baked at the equivalent spot. If you move the wheel or the driver seat offset `(0, 0.7, 0.35)`, re-derive grips and verify with a headless render. Tires have white sidewall rings (`wallMat` tori in `makeWheel`).

Items with a `count` (`triple` mushrooms, `tripleGreen` shells) deal a stack. **`tripleGreen` (type `tripleShell`) is stateful**: the first Space press calls `_activateTripleShell` → 3 shells orbit the kart (`ItemSystem.orbiters`, updated in `_updateOrbiters`: ram enemies / cancel incoming projectiles / shield the owner — the `kart._orbiter` back-ref). **Each ram, block, or fire consumes exactly one shell** (`sh.dead` flag → cleanup pass updates `itemCount`); pressing Space again `_fireOrbitShell`s one forward. Shells also all drop if the owner is hit/shrunk/falls. Lightning spawns 3D `_spawnBolt` meshes (`this.bolts`) and drops victims' orbiters; reusable effect presets live in `particles.js` (`shockwave`/`burst`/`itemPop`/`starBurst`/`shellTrail`/`boltFlash`/`boostRing`). Effects are intentionally flashy (big multi-layer explosion flash, twin-exhaust boost flames, white-cored drift sparks, a `boostRing` fired on every mini-turbo/mushroom/pad/draft from `kart.js`, plus `confetti` at the finish, `draftStreak` wind lines, `smoke` for the train, and theme-colored `dust(x,y,z,color)`); the pooled `ParticleSystem` is sized **920** in `game.js` so heavy moments don't starve — bump it (not the per-effect counts) if you add more simultaneous emitters. `ItemSystem.reset()` must dispose boxes/projectiles/orbiters/bolts (it routes meshes through `U.disposeObject`). **The Bob-omb is lobbed in a ballistic arc** (`_spawnBomb` sets `arc`+`vy`+`gravity`; in `update` it integrates `vy` under gravity instead of hugging the road, *skips* wall reflection so it flies over walls, and detonates the moment it lands — `vy<=0 && y<=groundY` — or on direct kart contact). `explodeAt(pos, radius, owner)` is **1.5× radius** and **never affects `owner`** (the thrower can't self-destruct). **The Spiny Shell** (`spiny`, type `spiny`, purple `0x9b30ff`) is a rare back-of-pack roll (added only to the two lowest `rollItem` tiers, weights 3/7 — rarer the higher you are) that **auto-targets 1st place**: `_spawnSpiny`/`_updateSpiny` drive it along the **track centerline** (lateral≈0, "the middle of the course") via an absolute arc-length cursor `_f = owner.progress·N` advanced at 85 u/s, recomputing the current leader (max `progress`, excluding the owner) every frame; when `_f` reaches the leader's arc-length it `_detonateSpiny`s on them (`launch` + 6-unit splash). Any **non-leader** kart it physically grazes en route is also crashed (`launch`) but does **not** consume the shell (the launched kart's `invulnTimer` keeps it from being multi-hit); **only hitting 1st removes the shell**. It **never expires on a timer** (60 s safety cap only) and, if the leader is star-invincible, hovers beside them until vulnerable — it doesn't vanish until it crashes 1st. Each `hz` exposes `hitPoints[]` (world `Vector3`s), `dangerous` (often time-gated by its motion cycle), `radius`, `effect` (`spin`|`launch`), and `markerPos`; meshes are built in the `Build` map and animated per-kind in `_animate`. Enemy meshes are **smooth-shaded** (`hazards.js` `M()` no longer sets `flatShading`) to match `concept_art/Mario_Kart_2.1.png`; designs follow that art (Koopa = ochre head + rounded beak + green shell/cream belly + **green boots**; Thwomp = blue-grey stone with a **spiked border all around** + furrowed brows + gritted teeth; Piranha = red bulb with white oval spots + white lip-ring + fangs; Goomba lives in `scenery.js` `Build.goomba()`, smooth dome + angry brows + frown/fangs). The **Snowman** (`snow`; replaced the old rolling "snowball") is a stacked-snowball figure (coal face, carrot nose, twig arms, top hat, scarf) that **waddles** along the road; the **Icicle** is enlarged + segmented with a frosted base; the **Comet** (`rainbow`) now **roams the whole track** (advances an arc-length cursor `_f` along the centerline at `travelSpeed`, with random lateral wander re-targeted each landing) while **bouncing in big parabolic arcs** — `dangerous` only near the ground (`hop<2.6`), with a landing-shadow telegraph — instead of crawling. **Lakitu** reuses the upgraded `MK.SceneryBuild.lakitu()` mesh (puffy cloud-with-face + Koopa rider: green shell, round orange-rim glasses, beak, arms — also the start-line signal official); the enemy orbits high (`center`/`alt`/`orbitR`/`orbitW`) above a mid-lap point, is itself **never `dangerous`**, and every `dropEvery` (~3.4s, capped at `maxDrops` simultaneous via `_countFallingKoopas`) lobs a **falling Koopa**: a dynamic `fallingKoopa` hazard pushed through `this._spawnQueue` (deferred to *after* the `update()` loop so the array isn't mutated mid-iteration), integrated under gravity with a telegraphing ground-shadow blob, **only `dangerous` while within ~2.6 of the ground** (so it can't hit karts from the sky), and after landing it lingers ~2.4s then sets `_remove`→`_needCull` so `update()` splices it out and frees its mesh/shadow via `U.disposeObject`. Collisions are gated by `kart.isHittable()` (Star/invuln immune). `nearestDangerAhead(kart, dist)` feeds the AI's avoidance nudge in `ai.js`. (Enemies are **not** shown on the minimap — `HUD.drawMinimap(karts, player)` plots karts only; the player marker is emphasized with a pulsing white halo, a heading triangle, a thick white rim and a white center dot, and is drawn last so it stays on top.) Built/updated/reset by `Game` alongside scenery/items. Add a kind: a `Build.*` mesh + an `_add` branch (set `radius`/`effect`/`hitPoints`) + an `_animate` case + an entry in the `build()` per-theme plan (or, for a singleton like Lakitu, a dedicated `_addX()` called from `build()`).
- **Difficulty**: `DIFF` in `game.js` scales **AI** `maxSpeed`/`accel` and skill; the player is unaffected.
- **AI edge-awareness** (`ai.js`): the controller aims a look-ahead target offset by a `lane`, clamped to a `safeLane` margin inside the road. On **`voidRespawn` courses (no walls, e.g. Rainbow Road)** it is deliberately conservative so CPUs stop driving off the edge: tighter `safeLane` (~0.52·roadHalf), shorter look-ahead, gentler apex, early+hard corner braking, **no drift while near an edge**, and a last-ditch recovery (brake + steer straight to center) when it detects lateral drift *outward* (`driftingOut`, derived from `this._prevLat`). It also skips the catch-up rubber-band boost there. Walled courses keep the original racing line. The player is never touched by any of this.

### Assets: procedural graphics/SFX + real MP3 BGM

- **Procedural (no files):** all textures are canvas-generated (`utils.js`: `questionTexture`, `checkerTexture`, sprite/particle textures), **sound effects and the speed-linked engine tone are synthesized at runtime** with the Web Audio API (`audio.js`), and every mesh is built from Three.js primitives. `client/public/assets/` is empty by design. Concept-art references for the art direction live in `concept_art/` (not loaded at runtime).
- **Memory release (per-race teardown):** every race builds fresh geometry/materials/**canvas textures**, so `cleanupRace` must free them or GPU memory leaks across races. All `reset()`/`dispose()` paths route their object tree through **`U.disposeObject(root)`** (`utils.js`), which disposes geometries, materials, **and every texture the materials hold** — `material.dispose()` alone does *not* free textures. `ParticleSystem.dispose()` frees the 460 pooled sprite materials + the shared particle texture. When you add anything that creates a texture or material in a race-scoped system, make sure it's reachable from that system's `reset()`.
- **Watch the light count (perf):** `MeshStandardMaterial` recompiles per active light, so a scene with dozens of `PointLight`s (Bowser's Castle once gave every scattered `castlePillar` its own light → ~20+) tanks the framerate. Pillars now use only their additive flame sprite for "glow"; the **castle keeps just the ~5 ambient lava `PointLight`s**. Don't reintroduce per-prop dynamic lights — fake it with emissive materials / additive sprites.
- **Set pieces & decoration** are all procedural meshes: themed **roadside fences/guardrails** follow the track in `track.js` (`_buildFences` → `_fencePost`/`_fenceRail`, keyed by `theme.props`), and per-course **background landmarks + props** are in `scenery.js` `Build.*` (e.g. `peachCastle`, `bowserKeep` + an encircling battlement ring with embedded `whompBlock` faces, `igloo`, ice-bergs, `castlePillar`, `ringedPlanet`, `flower`), placed in `_buildEnvironment`/`_pickProp`. Distant backdrops are course-specific helpers called from `_buildEnvironment`: `_buildGrassBackdrop` (layered rounded rolling hills + green mountains + Peach's castle + drifting `balloon`s), `_buildSnowBackdrop` (icy-blue snow-capped mountains + evergreen treeline + ice-bergs + `_buildSnowfall`), Rainbow's `_buildStarfield`+`_buildNebula` (twinkling colored stars + glowing nebula puffs), **`_buildBeachBackdrop`** (animated ocean disc + wet-sand ring around a 290-radius sand island, offshore island mounds with palms, an additive sun sprite, circling seagulls, and a `rockArch` straddling the road at f≈0.56) and **`_buildDesertBackdrop`** (two rings of strata-banded `mesa`s, a huge sun, circling vultures, and — when `course.train` exists — the **railway**: `MK.TrainPath.build()` samples the train loop, blends its height up to road level near crossings, then `_buildTrainTracks` lays a gravel ribbon, two rail ribbons, sleepers, and `crossbuck` signs at every road-entry transition; `_scatterProps` skips prop spots within 6 units of the rails via `_nearRail`). **Animated background** state lives on the `Scenery` instance and updates in `Scenery.update(dt, karts)`: `drifters` (balloons **and clouds** orbit+bob), `snowfall` (a `Points` flake field that follows the player), `birds` (gulls/vultures circle + flap `userData.wings`), `oceanTex` scroll, and star `_twinkle` — all freed via the normal `reset()`→`U.disposeObject` path.
- **Real BGM (`client/public/music/*.mp3`):** background music is no longer synthesized — it plays scene-appropriate MP3 tracks via HTML5 `Audio` elements (crossfaded, looped) in `audio.js`. The scene→track map is `MK.MUSIC` (`title`/`select`/`results`/`star`) in `audio.js`; per-course tracks are the `music` field on each course in `courses.js`. `Game`/`UI` drive it: `playMenuMusic(key)` (title/char-select/course-select/results), `playCourseMusic(course)` (race build), and a per-frame `playStarMusic()`/`endStarMusic()` overlay (`Game._updateStarMusic`) that swaps to the Star track while the player is invincible and reverts to the course track. **`setMusicRate(rate)`** changes `playbackRate` on the current and future tracks — `Game` sets `1.08` when the player enters the final lap (with a `FINAL LAP!` HUD flash) and resets to `1` at race build/finish. New synthesized SFX: `boostPad`, `draftWhoosh`, `trainWhistle`, `splashSfx`. Autoplay-policy is handled by retrying `.play()` on the first user gesture (`_attemptPlay`/`_kickPending`); `M` mutes everything, `B` toggles music (`toggleMusic`), pause uses `pauseMusic`/`resumeMusic`.
- **The `music/` MP3s are committed** (`client/public/music/*.mp3`, ~77 MB) — they are real Mario Kart 64 OST tracks and are tracked in the repo by project decision (they were previously gitignored; that exclusion was removed). Without them the game still runs (SFX work; menus/races are just silent). Code degrades gracefully if `Audio` is unavailable (headless) or a file is missing.

### Frame timing

Physics is frame-rate independent: `dt` is clamped (`≤0.045`) and smoothing uses exponential damping helpers `U.damp` / `U.dampAngle` / `U.dampFactor` rather than fixed lerp factors. `performance.now()` drives cosmetic animations (bobbing, flicker, star flash).
